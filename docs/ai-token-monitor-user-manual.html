1. 产品说明

AI Token 监控用于统计员工在 VS Code、Cursor、Copilot、Continue、Cline 等 AI 工具中的 Token 使用量，并将结果上报到公司服务器统一汇总。

系统由三部分组成：扩展负责控制台与状态展示，本地 ai-monitor.exe 负责 MITM 采集，服务端负责汇总与展示。

2. 使用对象

普通员工主要关心安装、登录、启用透明代理和查看链路状态。管理员除这些内容外，还需要掌握 upstreamProxy、证书安装、TUN 环境拦截和 Output 日志定位。

3. 安装前准备

  • Windows 10 或 11
  • 已安装 VS Code 或 Cursor
  • 能访问公司上报服务
  • 若已有公司代理或本地代理，请准备明确的代理地址
https://otw.tech:59889

4. 安装扩展

4.1 获取安装包

ai-token-monitor-win32-x64-0.1.43.vsix

4.2 在 VS Code 或 Cursor 中安装

  1. 打开命令面板 Ctrl+Shift+P
  2. 执行 Install from VSIX
  3. 选择 VSIX 文件
  4. 安装完成后重载窗口

5. 首次配置

安装完成后，打开左侧“腾轩 AI 监控”面板，配置服务端地址和身份信息。

  • 工号 / 用户 ID
  • 姓名
  • 部门

如果系统已启用登录注册，建议走登录流程，不要直接手工改写身份。

6. 配置项说明

配置项说明示例
aiTokenMonitor.serverUrl服务端地址https://otw.tech:59889
aiTokenMonitor.userId工号或唯一身份标识10001
aiTokenMonitor.userName姓名张三
aiTokenMonitor.department部门研发部
aiTokenMonitor.transparentMode是否启用透明代理接管true / false
aiTokenMonitor.proxyPort本地 MITM 代理端口18090
aiTokenMonitor.gatewayPort本地 API 网关端口18091
aiTokenMonitor.upstreamProxy上游代理地址socks5://127.0.0.1:10808
aiTokenMonitor.apiKey上报 API Key与服务端一致
留空的 upstreamProxy 是合法的；只有你明确依赖现有代理出网时，才建议手工填写它。

7. 推荐使用方式

无特殊网络环境时，推荐直接在面板中启用透明代理并重载窗口。有现有代理链路时，优先配置明确的 upstreamProxy，再启用透明代理。

Claude Code 如果是独立 CLI 运行，推荐先执行 ai-monitor.exe --launch-preset powershell，再在该终端内运行 claude；若 claude 已在 PATH 中，也可直接执行 ai-monitor.exe --launch claude。

8. 日常使用说明

配置完成后，通常只需正常使用 AI 工具即可。扩展会在后台采集请求、维护离线队列并同步 my-stats。

9. 监控面板说明

面板包含基础设置、身份状态、链路自检与统计概览。链路自检尤其重要，因为它能区分“本地代理未起”“服务端不可达”“身份冲突”“当前窗口未重载”等不同问题。

10. 自动更新说明

扩展会从服务端读取最新 VSIX 版本元数据并在发现新版本时提示安装。当前 Windows 最新版本为 0.1.43。

11. 本地客户端说明

Windows 发布版内置的 ai-monitor.exe 当前版本为 2.1.1。扩展启用透明代理时，会先把它复制到当前用户数据目录，再从该目录启动，避免直接从扩展目录执行时被拦截。

Output 面板中的 AI Token Monitor Proxy 会展示证书安装、启动命令、退出码和最近一次启动错误。

12. 常见场景指引

  • 只想统计，不想动现有网络：先不要开启透明代理。
  • 机器依赖公司代理出网：先填 upstreamProxy。
  • 同时装了 VS Code 与 Cursor：两边可同时使用，状态隔离。
  • 安装了 OpenVPN 但没连接：新版本不会因为未激活网卡误阻断。

13. 排障指南

13.1 服务端不可达

检查 serverUrl、网络连通性和服务端 /health。

13.2 身份冲突

同一工号已绑定其他姓名时，collect 会被 409 拒绝，需要管理员处理。

13.3 证书安装失败

查看 Output 是否包含 spawn error code、证书安装输出或本地 client 启动异常。

13.4 Takeover blocked

通常意味着存在真正启用中的高风险 TUN、PAC 环境无明确上游、或上游代理不可达。

14. 管理员分发建议

建议统一分发最新 Windows VSIX，并要求问题用户同时提供扩展版本、内置 client 版本、是否启用透明代理以及 Output 最后 30 行。

15. 发布与升级说明

服务端通过扫描扩展目录中的文件名来识别最新版本，例如：

ai-token-monitor-win32-x64-0.1.43.vsix

当前最新版本：

  • 扩展：0.1.43
  • 内置 client：2.1.1

16. FAQ

为什么服务端有数据，不等于链路正确？ 因为计数变化可能来自别的来源应用或旧实例，必须结合 Output 和链路自检判断。

为什么切换服务端后登录状态不沿用？ 因为认证会话按服务端地址和身份隔离，防止把旧 token 发到其他服务器。

17. 快速检查清单

  1. 确认扩展是不是最新版。
  2. 确认服务端地址是否正确。
  3. 确认工号和姓名是否完整。
  4. 执行链路自检。
  5. 查看 Output 面板 AI Token Monitor Proxy。
  6. 必要时把最后 30 行日志发给管理员。
