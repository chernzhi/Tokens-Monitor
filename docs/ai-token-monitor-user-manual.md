# AI Token 监控使用手册

版本：2026-04-13

适用范围：
- Windows 员工终端
- VS Code / Cursor 扩展使用者
- 需要排查本地代理、证书、上报链路的管理员

---

## 1. 产品说明

AI Token 监控用于统计员工在 VS Code、Cursor、Copilot、Continue、Cline 等 AI 工具中的 Token 使用量，并将结果上报到公司服务器统一汇总。

系统由三部分组成：
- VS Code / Cursor 扩展：负责界面、身份信息、链路自检、更新检查与本地代理接管。
- 本地客户端 `ai-monitor.exe`：负责启动 MITM 代理、安装当前用户证书、解析 AI 请求中的 token 用量。
- 服务端：负责接收上报、规范化 provider/source_app、汇总到个人和大屏。

你可以把它理解成：
- 扩展负责“控制台”
- `ai-monitor.exe` 负责“本地采集”
- 服务端负责“汇总与展示”

## 2. 使用对象

### 普通员工

主要需要完成：
- 安装扩展
- 登录或填写身份信息
- 按需启用透明代理
- 通过监控面板查看链路状态与上报结果

### 技术支持 / 运维同事

除以上内容外，还需要掌握：
- `upstreamProxy` 的使用条件
- 本地证书安装问题排查
- TUN / 虚拟网卡环境下为什么会阻止自动接管
- 如何判断“是采集失败，还是只是面板没显示”

## 3. 安装前准备

请先确认以下条件：
- 使用 Windows 10 或 Windows 11。
- 已安装 VS Code 或 Cursor。
- 本机可以访问公司上报服务。
- 若公司网络依赖本地代理、系统代理或桌面代理，请提前确认代理地址。

默认服务端地址通常为：

```text
https://otw.tech:59889
```

如果管理员给了其他地址，以实际地址为准。

## 4. 安装扩展

### 4.1 获取安装包

向管理员获取 `.vsix` 扩展包，例如：

```text
ai-token-monitor-win32-x64-0.1.43.vsix
```

### 4.2 在 VS Code 中安装

1. 打开 VS Code。
2. 按 `Ctrl+Shift+P` 打开命令面板。
3. 输入 `Extensions: Install from VSIX...`。
4. 选择收到的 `.vsix` 文件。
5. 安装完成后点击“重载窗口”。

### 4.3 在 Cursor 中安装

Cursor 的安装方式与 VS Code 基本相同：

1. 打开 Cursor。
2. 按 `Ctrl+Shift+P`。
3. 选择 `Install from VSIX`。
4. 选择 `.vsix` 文件。
5. 安装后重载窗口。

### 4.4 安装完成后的变化

安装成功后，你会看到：
- 左侧活动栏出现“腾轩 AI 监控”图标。
- 扩展启动后会自动初始化网络拦截器。
- 如果启用了透明代理，扩展会尝试准备本地代理与证书。

## 5. 首次配置

### 5.1 打开监控面板

点击左侧活动栏中的“腾轩 AI 监控”图标，进入监控面板。

### 5.2 配置服务端地址

面板中的“上报地址”决定数据发往哪里，默认一般是：

```text
https://otw.tech:59889
```

不要额外加尾部斜杠。

### 5.3 配置身份信息

如果系统尚未登录，请至少填写：
- 工号 / 用户 ID
- 姓名
- 部门

如果系统启用了登录注册，则建议使用登录方式，不要手工乱填，以免与服务端已有身份冲突。

### 5.4 登录与注册

常见流程：
- 已有账号：直接登录。
- 首次使用：注册后自动回填工号、姓名、部门。
- 账号存在但未设置密码：按面板提示设置密码。

登录成功后，扩展会记录与当前服务端、当前工号、当前姓名绑定的认证会话。后续如果你切换到别的服务端或别的账号，旧 token 不会再被错误复用。

## 6. 配置项说明

扩展设置项前缀为：

```text
aiTokenMonitor.*
```

### 6.1 `aiTokenMonitor.serverUrl`

含义：后端服务器地址，用于 collect、my-stats、更新检查等请求。

示例：

```text
https://otw.tech:59889
```

### 6.2 `aiTokenMonitor.userId`

含义：当前用户工号或唯一标识。

说明：
- 已登录时一般自动回填。
- 不建议随意修改。

### 6.3 `aiTokenMonitor.userName`

含义：姓名。

说明：
- 用于服务端身份校验。
- 与工号不一致时，collect 可能被服务端拒绝。

### 6.4 `aiTokenMonitor.department`

含义：部门名称。

### 6.5 `aiTokenMonitor.transparentMode`

含义：是否启用透明代理采集。

说明：
- 默认关闭。
- 开启后，扩展会尝试启动本地 `ai-monitor.exe`，安装当前用户证书，并把 VS Code / Cursor 的 `http.proxy` 指向本地代理。
- 若扩展判断当前环境风险高，则会阻止接管。

### 6.6 `aiTokenMonitor.proxyPort`

含义：本地 MITM 代理端口，默认：

```text
18090
```

### 6.7 `aiTokenMonitor.gatewayPort`

含义：本地 API 网关端口，默认：

```text
18091
```

### 6.8 `aiTokenMonitor.upstreamProxy`

含义：上游代理地址。

用途：
- 你的机器必须通过公司代理、系统代理、HTTP 代理或 SOCKS5 代理访问外网时，用这个字段让本地 MITM 的外联继续经过原代理。

合法格式：

```text
http://127.0.0.1:7890
https://proxy.company.com:8443
socks5://127.0.0.1:10808
```

注意：
- 留空表示不显式指定，由扩展自行判断是否复用 VS Code 代理、系统代理或环境变量。
- 不能填裸地址，例如 `127.0.0.1:7890`。
- 不能填空字符串以外的无效值。

### 6.9 `aiTokenMonitor.apiKey`

含义：服务端要求 API Key 认证时，扩展会在请求头中带上它。

### 6.10 `aiTokenMonitor.copilotOrg`

含义：GitHub Copilot 组织名，用于企业统计接口，不是日常上报的必填项。

## 7. 推荐使用方式

### 7.1 普通办公环境

如果你的机器没有特殊代理软件：
- 填好服务端和身份信息。
- 打开监控面板。
- 启用透明代理。
- 按提示重载窗口。

### 7.2 机器上已有公司代理或本地代理

如果你的机器依赖本地代理软件出网：
- 优先在 `upstreamProxy` 中显式填写可复用的 HTTP / HTTPS / SOCKS5 地址。
- 然后再启用透明代理。

这样可以避免扩展把原有链路错误改成直连。

### 7.3 同时使用 VS Code 与 Cursor

可以两边都安装扩展。

系统会按“服务端地址 + 用户 ID + 应用名”隔离本地状态，因此：
- VS Code 和 Cursor 的本地缓存互不覆盖。
- 两边都可以上报到同一账号。

### 7.4 Claude Code 与 CLI 模式

Claude Code 可以统计，但要区分两种情况：
- 如果它运行在 VS Code / Cursor 扩展宿主中，扩展拦截器可以直接统计。
- 如果它是独立 CLI 进程，则流量必须经过本地 `ai-monitor.exe` 才能统计。

推荐 CLI 用法：

```text
ai-monitor.exe --launch-preset powershell
```

然后在打开的 PowerShell 中执行：

```text
claude
```

如果 `claude` 命令已经在 PATH 中，也可以直接：

```text
ai-monitor.exe --launch claude
```

这样 Claude CLI 的请求会自动进入本地 MITM，从而被记录到统计链路中。

## 8. 日常使用说明

安装并配置完成后，日常使用通常不需要额外操作。

你只需要：
- 正常使用 Copilot、Cursor、Continue、Cline 等 AI 工具。
- 在需要时打开监控面板查看统计与链路状态。

扩展会自动做这些事：
- 捕获 AI 请求。
- 将 token 用量写入本地待发送队列。
- 后台尝试发送到服务端。
- 同步 my-stats 到面板。
- 检查是否有扩展新版本。

## 9. 监控面板说明

监控面板主要分为几块。

### 9.1 基础设置

用于查看或修改：
- 服务端地址
- 工号
- 姓名
- 部门
- 上游代理

### 9.2 身份状态

面板会检查当前工号与姓名在服务端是否匹配，常见状态：
- 正常：当前工号与服务端记录一致。
- 新用户：服务端尚无该工号，首次上报会自动创建。
- 冲突：工号已绑定其他姓名，collect 会被拒绝。
- 不可用：当前无法连接服务端或身份检查接口不可用。

### 9.3 链路自检

链路自检会展示：
- 身份校验
- 本地代理
- 代理环境
- 证书安装
- 上游代理
- 最近上报

这是最重要的排障入口。

### 9.4 统计信息

通常会展示：
- 今日 tokens
- 今日请求数
- 服务端 overview 概览
- 来源、应用、模型维度的拆分

## 11. 本地客户端 `ai-monitor.exe` 说明

扩展在透明代理模式下会自动调用内置的 `ai-monitor.exe`。

它负责：
- 启动本地 MITM 代理
- 安装当前用户证书
- 捕获并解析请求响应
- 上报 token 用量

当前 Windows 发布版内置 client 版本为：

```text
2.1.1
```

### 11.1 它安装在哪里

扩展首次启用透明代理时，会优先把内置二进制复制到当前用户数据目录，再从该目录启动，避免直接从扩展安装目录执行时被文件锁或安全软件拦截。

### 11.2 它什么时候会启动失败

常见原因：
- 证书安装被系统策略或权限阻止。
- 本地代理端口被占用。
- 上游代理配置错误。
- 当前存在真实启用中的高风险 TUN / 虚拟网卡环境。

### 11.3 如何看它的日志

在 VS Code / Cursor 中打开 Output 面板，选择：

```text
AI Token Monitor Proxy
```

这里会显示：
- 启动命令
- 证书安装结果
- 本地代理 stdout/stderr
- 退出码
- 最近一次启动错误

## 12. 常见场景指引

### 12.1 只想统计，不想影响现有网络

做法：
- 默认不要启用透明代理。
- 先观察扩展拦截器是否已能采到部分请求。
- 只有在需要更完整采集时，再启用透明代理。

### 12.2 公司内网机器必须走代理

做法：
- 明确本机现有代理类型。
- 如果已有 HTTP / SOCKS5 本地端口，直接填到 `upstreamProxy`。
- 再启用透明代理。

### 12.3 机器上安装了 OpenVPN、Clash、sing-box、WireGuard

说明：
- 如果这些高风险虚拟网卡当前处于 `Up` 状态，扩展可能主动阻止接管，防止把现有链路改坏。
- 如果只是安装了网卡但当前未启用，新版本不会再因为它们而误阻断。

### 12.4 首次安装后本地代理起不来

先看 Output 面板里的：
- `Failed to start local proxy process`
- `Failed to install current-user CA certificate automatically`
- `Takeover blocked`

不同日志对应不同问题，不要只看面板颜色。

## 13. 排障指南

### 13.1 面板显示“服务端不可达”

检查：
- `serverUrl` 是否正确。
- 本机是否能访问该地址。
- 服务端 `/health` 是否返回正常。

### 13.2 面板显示“工号与姓名冲突”

说明：
- 服务端已存在同工号但不同姓名的绑定记录。

处理：
- 不要继续手工改名字硬试。
- 联系管理员修正身份绑定。

### 13.3 面板显示“当前窗口仍待重载”

说明：
- 本地代理可能已启动，但当前窗口内既有网络连接还没有切到新链路。

处理：
- 直接重载窗口。

### 13.4 Output 中看到证书安装失败

常见原因：
- 当前用户证书存储被策略限制。
- 安全软件拦截。
- 本地 client 启动失败。

处理顺序：
1. 看 Output 中是否有更具体的 `spawn error code`。
2. 确认内置 `ai-monitor.exe` 版本是否是最新。
3. 再判断是否为权限或安全软件问题。

### 13.5 Output 中看到 `upstream_proxy 无效`

说明：
- 上游代理配置值格式错误。

正确示例：

```text
http://127.0.0.1:7890
socks5://127.0.0.1:10808
```

错误示例：

```text
127.0.0.1:7890
proxy.company.com:8080
```

### 13.6 Output 中看到 `Takeover blocked`

说明：
- 扩展为了避免破坏当前网络环境，主动停止接管。

常见触发原因：
- 当前启用了高风险 TUN 虚拟网卡。
- 系统是 PAC 自动代理但没有明确可复用上游。
- 上游代理不可达。

处理方式：
- 按 Output 建议配置明确上游代理。
- 或保持当前不接管，仅使用非透明采集能力。

## 16. FAQ

### Q1：为什么安装后不建议默认立即接管网络？

因为不同员工机器的代理环境差异很大。默认关闭透明代理更安全，避免与现有桌面代理、系统代理、PAC、TUN 环境直接冲突。

### Q2：为什么大屏有数据，但我怀疑不是正确链路？

因为服务端计数增加不等于你期望的链路已经正确工作。必须结合来源应用、请求路径、Output 日志和面板链路自检一起判断。

### Q3：为什么已经安装了 OpenVPN 但没连接，也会影响监控？

新版本已经修正这一点。只有当前状态为 `Up` 的高风险虚拟网卡才会阻断透明代理接管。

### Q4：为什么我切换了服务端后，之前的登录状态没有继续生效？

这是设计行为。认证会话会按“服务端地址 + 工号 + 姓名”绑定，避免把旧 token 错发到其他服务器。

## 17. 附录：快速检查清单

普通用户最常用的排查步骤：

1. 确认扩展版本是不是最新。
2. 确认服务端地址是否正确。

4. 在监控面板执行链路自检。
5. 如果启用了透明代理，查看 Output 面板中的 `AI Token Monitor Proxy`。
6. 如果仍有问题，把最后 30 行日志发给管理员。
